# ivllm Implementation Reference

This document describes how the `isambard-compute-tunnel` pattern is implemented in the `isambard-vllm` project (`ivllm` CLI). It serves as a concrete, working example in TypeScript + Bash for reference when implementing the pattern in another project or language.

Source repository: `ai4ci/isambard-vllm`

---

## SLURM Script Side

The inference SLURM script is generated by `src/templates/inference.ts`.

### Compute hostname discovery

```bash
COMPUTE_HOSTNAME=$(hostname)
```

Called immediately on job start, before any other work. The result is written into the rendezvous file.

### Rendezvous file: initial write

```bash
JOB_DETAILS="${workDir}/job_details.json"

jq -n \
  --arg status "initialising" \
  --arg job_name "${jobName}" \
  --arg slurm_job_id "$SLURM_JOB_ID" \
  --arg compute_hostname "$COMPUTE_HOSTNAME" \
  --arg model "${model}" \
  --argjson server_port ${serverPort} \
  '{status: $status, job_name: $job_name, slurm_job_id: $slurm_job_id,
    compute_hostname: $compute_hostname, model: $model, server_port: $server_port}' \
  > "$JOB_DETAILS"
```

The local client writes a `pending` version of this file before `sbatch` runs, so the file always exists. The job overwrites it with `initialising` (including `compute_hostname`) as soon as it starts.

### Status update to `running`

After the health check loop confirms vLLM is serving:

```bash
jq '.status = "running"' "$JOB_DETAILS" > "$JOB_DETAILS.tmp" && mv "$JOB_DETAILS.tmp" "$JOB_DETAILS"
```

The `tmp`+`mv` pattern is used throughout to make writes atomic — this prevents the local client from reading a half-written file.

### Exit trap for diagnostics

```bash
on_exit() {
  EXIT_CODE=$?
  trap - EXIT
  CURRENT_STATUS=$(jq -r '.status' "$JOB_DETAILS" 2>/dev/null || echo "unknown")
  if [ "$CURRENT_STATUS" = "initialising" ] || [ "$CURRENT_STATUS" = "running" ]; then
    jq '.status = "failed" | .error = "job exited unexpectedly"' \
      "$JOB_DETAILS" > "$JOB_DETAILS.tmp" && mv "$JOB_DETAILS.tmp" "$JOB_DETAILS"
  fi
  persist_slurm_accounting
  exit $EXIT_CODE
}
trap on_exit EXIT
```

`persist_slurm_accounting` saves `sacct` output to `slurm-accounting.txt` in the work directory. This survives job cancellation and is retrieved by the local client for crash diagnostics.

## Local Client Side

The local daemon is implemented in `src/commands/start.ts` (TypeScript, runs with Bun).

### Pre-flight

```typescript
const { stdout: ok } = await ops.runRemote("echo ok", { silent: true });
if (ok.trim() !== "ok") throw new Error("SSH pre-flight failed");
```

Simple echo test to the login node. Fails fast if SSH key auth is broken.

### Writing the pending lockfile

```typescript
await ops.runRemote(
  `set -C; echo '{"status":"pending","job_name":"${jobName}"}' > ${remoteWorkDir}/job_details.json`,
  { silent: true }
);
```

`set -C` (noclobber) makes the shell fail if the file already exists. If this fails, a session is already running with that job name.

### Submitting the job

```typescript
const { stdout: sbatchOut } = await ops.runRemote(`sbatch ${remoteScriptPath}`);
const match = sbatchOut.match(/Submitted batch job (\d+)/);
jobId = match![1];
```

### Polling the rendezvous file (5s interval — plain file read)

```typescript
const POLL_INTERVAL_MS = 5_000;

while (true) {
  await sleep(POLL_INTERVAL_MS);

  const { stdout: raw } = await ops.runRemote(
    `cat ${remoteWorkDir}/job_details.json 2>/dev/null || echo '{"status":"missing"}'`,
    { silent: true }
  );
  const details: JobDetails = JSON.parse(raw.trim());

  if (details.status === "running") {
    await onRunning(details);
    return;
  }
  if (details.status === "failed" || details.status === "timeout") {
    // handle failure...
  }
  // throttle squeue/sacct calls to ≤ once/60s
  if (Date.now() - lastSlurmPollTime >= SLURM_POLL_INTERVAL_MS) {
    lastSlurmPollTime = Date.now();
    const queueState = await getSlurmQueueState(config, jobId);
    // ...report queue position...
  }
}
```

### SSH tunnel

```typescript
function spawnTunnel(
  config: SshConfig,
  localPort: number,
  computeHost: string,
  remotePort: number
): ChildProcess {
  return spawn("ssh", [
    "-N",
    "-o", "BatchMode=yes",
    "-o", "ServerAliveInterval=10",
    "-o", "ServerAliveCountMax=3",
    "-o", "ExitOnForwardFailure=yes",
    "-L", `${localPort}:${computeHost}:${remotePort}`,
    `${config.user}@${config.loginHost}`,
  ]);
}
```

The tunnel process is stored and killed in the shutdown handler.

### Heartbeat

```typescript
const HEARTBEAT_INTERVAL_MS = 15_000;

heartbeatTimer = setInterval(async () => {
  try {
    const res = await fetch(`http://localhost:${localPort}/health`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
  } catch (e) {
    if (!shuttingDown) {
      console.error(`\nHeartbeat failed: ${(e as Error).message}`);
      await shutdown("heartbeat failure", 1);
    }
  }
}, HEARTBEAT_INTERVAL_MS);
```

Heartbeat polls the local end of the tunnel. If the service dies or the tunnel drops, the heartbeat fails and triggers a clean shutdown.

### Shutdown handler

```typescript
let shuttingDown = false;

async function shutdown(reason: string, code = 0): Promise<void> {
  if (shuttingDown) return;
  shuttingDown = true;

  console.log(`\nShutting down: ${reason}`);

  clearInterval(heartbeatTimer);

  if (tunnel) {
    tunnel.kill();
    tunnel = null;
  }

  if (jobId) {
    try {
      await ops.runRemote(`scancel ${jobId}`, { silent: true });
    } catch {}
  }

  process.exit(code);
}

process.on("SIGINT", () => shutdown("SIGINT"));
process.on("SIGTERM", () => shutdown("SIGTERM"));
```

The `shuttingDown` guard prevents double-shutdown from concurrent signals or heartbeat failure.

## Slurm Polling Throttle

```typescript
const SLURM_POLL_INTERVAL_MS = 60_000; // 60 seconds — Isambard AUP
let lastSlurmPollTime = 0;

// In the poll loop:
if (Date.now() - lastSlurmPollTime >= SLURM_POLL_INTERVAL_MS) {
  lastSlurmPollTime = Date.now();
  // call squeue or sacct here
}
```

The plain SSH file reads (`cat job_details.json`, `tail slurm.log`) run every 5 seconds and are not throttled — they don't touch the Slurm scheduler.

## Job Config Store

ivllm also implements a config store that persists the vLLM YAML config by job name:

```typescript
// ~/.config/ivllm/<jobName>.yaml
export function jobConfigPath(jobName: string): string {
  return path.join(homedir(), ".config", "ivllm", `${jobName}.yaml`);
}

export function saveJobConfig(jobName: string, src: string): void {
  mkdirSync(path.dirname(jobConfigPath(jobName)), { recursive: true });
  copyFileSync(src, jobConfigPath(jobName));
}
```

First run: `ivllm start qwen36 --config /path/to/qwen36.yaml` (saves config).
Subsequent runs: `ivllm start qwen36` (loads from `~/.config/ivllm/qwen36.yaml`).

This pattern is useful for any project where a per-job config file is needed and the user shouldn't have to re-specify it every run.

## Key Files in the Project

| File | Role |
|------|------|
| `src/commands/start.ts` | Local daemon (submit, poll, tunnel, heartbeat, shutdown) |
| `src/templates/inference.ts` | SLURM script generator (rendezvous file, health check, exit trap) |
| `src/slurm.ts` | `getSlurmQueueState` (squeue), `pollJobStatus` (sacct) |
| `src/ssh.ts` | `spawnTunnel`, `runRemote` helpers |
| `src/vllm-config.ts` | Config parsing, stripping, and job config store |
